## 候选 1: OpenViking

- **仓库**: https://github.com/volcengine/OpenViking
- **Stars**: 25,956 | **最近更新**: 2026-06-23 | **License**: AGPLv3
- **描述**: Open-source context database for AI Agents; file system paradigm; unified memory/resources/skills

### 解决的问题

传统 RAG 的向量存储是扁平的，缺乏全局视图，Agent 长任务产生的上下文无法有效管理。OpenViking 用"文件系统范式"统一组织 Agent 的记忆、资源、技能，支持 L0/L1/L2 三级按需加载和目录递归检索。

### 当前库映射位置

- Personal Wiki 目前使用 Obsidian + SQLite FTS5 + 向量检索；OpenViking 的"文件系统范式"和"三级加载"可以直接映射到 Wiki 的 note → section → chunk 层级。
- Personal OS 的 /api/agent/context 需要 hot/warm/cold 分层；OpenViking 的 L0/L1/L2 是现成的分层参考实现。
- 差异：OpenViking 是 Python 为主，Personal OS/Wiki 是 Node.js/TypeScript + Next.js 栈。不能直接嵌入，只能借鉴设计。

### 验证命令/方式

```bash
# 最小验证：本地安装并运行示例
pip install openviking --upgrade
python -c "import openviking; print(openviking.__version__)"
# 参考文档运行 quick-start 示例
# https://github.com/volcengine/OpenViking/blob/main/docs/en/quick-start.md
```

### 吸收/拒绝结论

**思路借鉴（拒绝直接代码吸收）**

理由：
1. 技术栈不匹配（Python vs Node.js/TS）
2. AGPLv3 传染性过强，不适合作为库依赖
3. 25K+ stars 证明设计方向正确，但「文件系统范式」和「L0/L1/L2 三级上下文」更适合作为设计参考写入 Wiki 架构决策

### 后续任务

- 任务：将 OpenViking 的 L0/L1/L2 分层设计写入 Personal Wiki 架构手册
- 动作：对比现有 /api/agent/context 的分层逻辑，提出 hot/warm/cold → L0/L1/L2 映射建议
- 验收：Wiki 中存在 OpenViking-设计借鉴.md，包含映射表和差异化分析

## 候选 2: agentmemory

- **仓库**: https://github.com/rohitg00/agentmemory
- **Stars**: 23,797 | **最近更新**: 2026-06-22 | **License**: MIT (推断)
- **描述**: Persistent memory for AI coding agents; built on iii engine; 95.2% R@5; 0 external DBs; 53 MCP tools

### 解决的问题

AI 编码 Agent（Claude Code、Hermes、OpenClaw 等）每次启动都是白板，需要重复解释上下文。agentmemory 提供持久记忆、本地 markdown 存储、知识图谱、混合检索和 MCP 工具集成。

### 当前库映射位置

- 直接映射到 Personal OS 的 Agent 记忆层：.agent-runs/<task-id>/ 的自动归档、上下文写回、任务链记忆。
- agentmemory 的 "local markdown storage" 与 Personal Wiki 的 Obsidian vault 高度一致。
- 它的 "MCP server" 模式可以直接被 Hermes 通过 MCP 调用，补充 Personal OS 的上下文能力。
- 差异：agentmemory 是面向编码 Agent 的单个用户工具，Personal OS 是面向多项目、多 Agent 的任务管理系统。需要评估是否能作为 MCP 插件接入，而不是核心库替换。

### 验证命令/方式

```bash
# 最小验证：安装 npm 包并启动 MCP 服务
npm install -g @agentmemory/mcp
# 或
npx @agentmemory/mcp --version
# 检查 README 中的 MCP 配置，接入 Claude Code / Hermes 做端到端测试
```

### 吸收/拒绝结论

**MCP 插件级吸收（部分吸收）**

理由：
1. 技术栈部分匹配（Node.js/npm），MCP 接口可直接对接 Hermes
2. 0 外部数据库依赖，与 Personal OS 的 SQLite 本地优先哲学一致
3. 但核心定位差异：agentmemory 是单个 Agent 的记忆工具，Personal OS 是项目级任务中枢。不宜替换核心，宜作为 MCP 能力扩展

### 后续任务

- 任务：在 Hermes 环境中安装并验证 @agentmemory/mcp 的可用性
- 动作：配置 MCP 接入，测试 Personal OS task 上下文能否通过 agentmemory MCP 持久化
- 验收：运行一次 Personal OS task 闭环后，agentmemory 中能找到对应 task 的上下文记录

## 候选 3: LOOP

- **仓库**: https://github.com/andrewowest/LOOP
- **Stars**: 0 | **最近更新**: 2026-04-27 | **License**: 未明确
- **描述**: A cognitive architecture for AI memory. Three-tier hierarchy with Bayesian weighting.

### 解决的问题

大多数 AI Agent 把记忆当作事后想法：把所有内容丢进向量数据库然后希望检索有效。LOOP 用认知科学启发的三层记忆（工作记忆 → 联想记忆 → 长期记忆）和贝叶斯重要性追踪，让 Agent 学会什么重要、什么该遗忘。

### 当前库映射位置

- 直接映射到 Personal OS /api/agent/context 的 tiers 设计：hot = WorkingMemory, warm = AssociativeMemory, cold = LongTermMemory。
- Controller 的 "attention and energy across memory tiers" 可以启发 Agent 自驱执行器的优先级分配逻辑。
- MemoryConsolidator 的自动升级管道（working → associative → long-term）对应 Personal Wiki 的 Inbox → Task → Archive / Knowledge 流程。
- 差异：LOOP 是研究型 Python 项目，0 stars，成熟度低，但设计思想清晰。

### 验证命令/方式

```bash
# 最小验证：本地安装并运行示例
pip install -e .
python -c "from loop_core import WorkingMemory, WorkingMemoryConfig; print('import ok')"
# 运行 README 中的 quick-start 示例
```

### 吸收/拒绝结论

**设计思路借鉴（拒绝直接代码吸收）**

理由：
1. 0 stars，成熟度不足，无法作为生产依赖
2. Python 栈，与现有 Node.js 系统不匹配
3. 但「三层记忆 + 贝叶斯重要性 + 自动合并」是高质量的架构参考，应写入 Wiki 作为 Agent 记忆设计依据

### 后续任务

- 任务：将 LOOP 的三层记忆架构写入 Personal Wiki 设计参考
- 动作：提取 WorkingMemory / AssociativeMemory / LongTermMemory 接口，映射到 Personal OS 的 Inbox/Task/Archive 状态机
- 验收：Wiki 中存在 LOOP-设计借鉴.md，包含贝叶斯重要性评分在任务优先级中的映射建议
