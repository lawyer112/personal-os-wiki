# 工作手册

这是这个仓库后续产品工作的对齐手册。

## 核心目标

Personal OS + Personal Wiki 不是更漂亮的收藏夹，也不是更漂亮的笔记软件。它要形成这个闭环：

```text
原始输入 -> 长期知识 -> 有人负责的工作 -> 证据 -> 复核 -> 提醒
```

Personal OS 管执行状态。Personal Wiki 管长期知识。Agent 把两者连接起来。

## 职责分工

| 表面 | 负责什么 | 不该做什么 |
| --- | --- | --- |
| 用户采集 | 低摩擦接收原始输入。 | 让用户分类、起标题、摘要、打标签、拆任务。 |
| Personal OS | 保存来源、工作状态、任务归属、复核、提醒。 | 把自己伪装成推理引擎。 |
| Personal Wiki | 保存可读的长期知识、反链、标签、概念和来源证据。 | 变成任务队列。 |
| Agent | 识别、抓取、总结、分类、写 Wiki、建任务、提交证据。 | 默认对每条被动采集都悄悄花 token。 |

## 采集规则

采集入口只接收一个原始入口：一个 URL 或一段短文本。浏览器采集、Telegram 转发、分享扩展、
未来浏览器插件都应该遵守这个规则。

不好的采集体验：

```text
URL + 标题 + 选中文本 + 备注 + 标签 + 任务字段
```

好的采集体验：

```text
粘贴或拖入一个链接。
```

后续由 Agent 判断这个链接来自博客、X、小红书、抖音、YouTube、GitHub、媒体网站还是普通网页。

## 处理规则

处理频率是策略，不是产品法律。实时、批处理、每天处理、纯手动都可以。仓库文档应该把它写成
可配置选择，而不是硬编码一个统一扫描间隔。

## 演示规则

公开演示必须先展示最简单的人类动作。如果真实用户只需要粘贴一个链接，演示里就不应该让用户输入
标题、选中文本、摘要、标签或任务字段。

所有公开演示数据必须是假数据。不要发布真实私有工作链接、聊天、token、主机名、局域网拓扑、
vault 内容或任务历史。
