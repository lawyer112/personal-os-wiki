# Repository Permissions

This repository is public, but write access is intentionally narrow.

## Roles

| Role | Access | Intended Use |
| --- | --- | --- |
| Public user | Read, fork, open issues, open pull requests | Try the project, report problems, propose changes. |
| Triage collaborator | Triage issues and discussions only | Label, reproduce, and route work without code write access. |
| Maintainer | Push branches, review pull requests, manage issues | Maintain code and docs through pull requests. |
| Admin / owner | Repository settings, releases, secrets, branch protection | Emergency control and release governance. |

Only trusted maintainers should receive write access. Admin access should stay
limited to the owner unless there is a real operational need.

## Branch Model

- `main` is the release branch.
- Contributors work through forks or feature branches.
- Pull requests are the normal merge path.
- CI must pass before merge.
- Force pushes and branch deletion on `main` should stay disabled.
- Squash merge is preferred for external contributions.

## GitHub Actions

- Workflow token permission is read-only by default.
- CI does not require repository secrets.
- Pull requests from forks should not receive write tokens or private secrets.
- If a future workflow needs publishing permissions, put it in a separate
  release workflow with explicit permissions.

## Secrets

This public repository should not contain production secrets. Runtime tokens
belong in each user's private deployment environment.

Never add:

- live `.env` files
- Personal OS or Personal Wiki tokens
- SSH keys or deploy keys
- private vaults, database dumps, logs, screenshots, or server inventories

## Recommended Repository Settings

- Visibility: public.
- Issues: enabled.
- Wiki: disabled; docs live in the repository.
- Discussions: optional, enable only if there is enough community traffic.
- Merge commits: disabled.
- Rebase merges: disabled.
- Squash merges: enabled.
- Delete head branches after merge: enabled.
- Branch protection on `main`: require CI status checks, block force pushes,
  block deletions, and keep direct pushes limited to admins/maintainers.

## Release Rule

Do not publish from the private live service tree. Public releases should be
made from this clean repository or from a clean export generated by
`scripts/prepare-public-repo.ps1`.
